## Context

当前项目是一个基于 U-Net++ 的道路车辆语义分割系统。项目根目录包含：
- 5 个 Markdown 文档（不含 README.md）
- 2 个 Shell 脚本
- 2 个测试文件
- 3 个推理脚本
- 1 个训练脚本
- 1 个配置目录
- 1 个源代码目录 (src/)

问题：
1. 文档散落在根目录，难以区分项目入口文档和详细文档
2. 脚本与 Python 代码混杂
3. 测试文件不在标准位置
4. 推理和训练脚本与核心源码分离
5. 缺少清晰的模块划分

## Goals / Non-Goals

**Goals:**
- 建立标准的 Python 项目目录结构
- 按功能分类组织文件（文档、脚本、测试、源码）
- 保持向后兼容，提供便捷的入口点
- 更新所有相关引用和文档

**Non-Goals:**
- 不修改任何代码逻辑
- 不添加新功能
- 不重构现有模块结构
- 不修改数据集和输出目录位置

## Decisions

### D1: 文档目录结构

**决定**: 创建 `docs/` 目录，移动所有非入口文档

**理由**:
- 遵循 Python 项目惯例
- 保持根目录整洁
- README.md 作为项目入口保留在根目录

**替代方案**:
- 所有文档放在根目录 → 当前状态，混乱
- 使用 `documentation/` → 名称较长，`docs/` 更常见

### D2: 脚本目录结构

**决定**: 创建 `scripts/` 目录存放 Shell 脚本

**理由**:
- Shell 脚本不是 Python 模块，不应在 src/ 中
- 便于用户快速找到辅助脚本
- 可以添加 `scripts/README.md` 说明

**替代方案**:
- 放在根目录 → 当前状态，混乱
- 放在 `tools/` 目录 → 与 `scripts/` 功能相似，选择更常见的命名

### D3: 测试目录结构

**决定**: 创建 `tests/` 目录，采用标准 pytest 结构

**理由**:
- pytest 默认查找 `tests/` 目录
- 符合 Python 测试最佳实践
- 便于添加更多测试文件

**替代方案**:
- 放在 `src/tests/` → 与源码混合
- 放在根目录 → 当前状态，不标准

### D4: 推理脚本位置

**决定**: 移动到 `src/inference/` 目录

**理由**:
- 推理脚本是核心功能的一部分
- 与 `src/models/`, `src/data/` 形成完整的模块结构
- 便于导入和复用

**替代方案**:
- 保留在根目录 → 不符合模块化设计
- 放在 `scripts/` → 这些是 Python 模块，不是辅助脚本

### D5: 训练脚本位置

**决定**: 移动到 `src/training/` 目录

**理由**:
- 训练是核心功能
- 与推理脚本形成对称结构
- 便于未来添加训练相关工具

**替代方案**:
- 保留在根目录 → 不符合模块化设计
- 与推理脚本合并 → 功能独立，应分开

### D6: 保持向后兼容

**决定**: 在根目录创建入口脚本代理

**理由**:
- 用户可能习惯了旧的命令路径
- 提供平滑的迁移路径
- 便于快速使用

**实现**: 创建 `train.py` 和 `inference.py` 代理脚本，导入并执行实际模块

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 用户习惯旧路径 | 在根目录创建代理脚本，更新文档说明新路径 |
| IDE 需要重新索引 | 提醒用户重新打开项目或刷新索引 |
| CI/CD 路径变更 | 更新相关配置文件（如有） |
| import 路径变更 | 更新所有脚本内部的 import 语句 |

## Migration Plan

1. 创建新目录结构
2. 移动文件到新位置
3. 更新 import 语句
4. 创建根目录代理脚本
5. 更新文档（README.md, CLAUDE.md）
6. 更新 Shell 脚本中的路径
7. 验证所有功能正常

**回滚策略**: 使用 git 记录所有变更，可随时回滚
