## Context

当前项目是一个道路车辆语义分割系统，包含 Flask 后端 API 和 Vue3 前端。系统当前存在严重安全漏洞：

- 认证系统完全无效（硬编码 token，不验证凭据）
- 模型文件加载存在远程代码执行风险（`torch.load(..., weights_only=False)`）
- CORS 配置允许任何来源访问
- 敏感信息硬编码和泄露
- 缺少速率限制和输入验证

约束条件：
- 需保持向后兼容，现有 API 接口不变
- 前端改动最小化
- 支持开发和生产环境的不同配置

## Goals / Non-Goals

**Goals:**
- 实现有效的 JWT 认证系统
- 修复模型文件不安全加载问题
- 加固 API 安全（CORS、速率限制、输入验证）
- 增强文件上传安全
- 移除敏感信息泄露
- 更新存在漏洞的依赖

**Non-Goals:**
- 不实现完整的用户管理系统（用户注册、权限管理）
- 不改变现有 API 接口路径
- 不重构整体架构

## Decisions

### 1. 认证方案：JWT Token

**选择**: 使用 JWT (JSON Web Token) 进行认证

**理由**:
- 无状态，适合 API 服务
- Flask 生态成熟支持（PyJWT、Flask-JWT-Extended）
- 前端改动小，只需在请求头添加 Authorization

**替代方案**:
- Session 认证：需要服务端存储，扩展性差
- OAuth2：过度设计，当前不需要第三方登录

### 2. 模型安全加载：weights_only=True + 白名单

**选择**: 强制使用 `weights_only=True`，仅加载张量数据

**理由**:
- 从根本上杜绝 pickle 反序列化漏洞
- PyTorch 2.0+ 推荐的安全加载方式

**替代方案**:
- safetensors 格式：需要转换所有现有模型，迁移成本高
- 模型签名验证：实现复杂，维护成本高

### 3. 速率限制：Flask-Limiter

**选择**: 使用 Flask-Limiter 实现速率限制

**理由**:
- Flask 原生支持
- 支持多种存储后端（内存、Redis）
- 可按 IP、用户、端点灵活配置

### 4. 文件上传安全：Magic number + 唯一命名

**选择**:
- 使用 python-magic 验证文件真实类型
- 使用 UUID 重命名上传文件
- 限制文件大小和数量

**理由**:
- Magic number 验证可防止扩展名伪造
- UUID 命名防止文件覆盖和路径遍历
- 大小限制防止存储耗尽

### 5. 配置管理：环境变量优先

**选择**: 所有敏感配置通过环境变量注入

**理由**:
- 12-Factor App 最佳实践
- 避免敏感信息硬编码
- 支持多环境部署

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| JWT Token 泄露 | 设置较短过期时间（1小时），提供刷新机制 |
| weights_only=True 导致旧模型无法加载 | 提供模型转换脚本，文档说明迁移步骤 |
| 速率限制误伤正常用户 | 设置合理阈值，提供白名单机制 |
| Magic number 验证增加依赖 | python-magic 是轻量级库，系统依赖 libmagic |
| 环境变量未设置导致服务无法启动 | 提供默认值（仅开发环境），生产环境强制检查 |

## Migration Plan

### 阶段一：紧急修复（立即）
1. 修复 `torch.load` 的 `weights_only=False` 问题
2. 设置安全的 SECRET_KEY 默认值检查
3. 关闭生产环境 DEBUG 模式

### 阶段二：认证系统（1-2天）
1. 实现 JWT 认证中间件
2. 保护 DELETE 等敏感端点
3. 前端集成 token 管理

### 阶段三：API 加固（2-3天）
1. 配置 CORS 白名单
2. 实现速率限制
3. 增强文件上传验证
4. 移除敏感信息泄露

### 阶段四：依赖更新（1天）
1. 更新 requirements.txt
2. 测试兼容性
3. 更新文档

**回滚策略**:
- 每个阶段独立提交，可单独回滚
- 保留旧认证代码路径，通过配置开关切换
